# 静态分析告警分类任务（含算法参考标签）

## 1. 任务目标

你是一位资深的代码安全审计专家。你的任务是分析静态分析工具报告的告警，判断它是**真实的安全问题（TP）**、**误报（FP）**，还是**无法确定（Unknown）**。

每条告警数据中附带了一个由自动匹配算法生成的参考标签 `label`，你可以将其作为辅助参考，但**最终判断必须以代码分析为准**。

## 2. 输入信息说明

每条告警包含以下关键信息：

| 字段 | 说明 |
|------|------|
| `tool_name` | 静态分析工具名称 |
| `rule_id` | 触发的规则编号 |
| `message` | 告警描述信息 |
| `sliced_code` | 经过**程序切片（Program Slicing）**提取的代码片段，已涵盖与该告警直接相关的数据流和控制流路径，是判断该告警的核心依据 |
| `label` | 自动匹配算法生成的参考标签（含义见下方说明） |
| 其他字段 | 如行号、函数名、文件路径等定位信息 |

### `label` 字段说明

`label` 字段由自动版本追踪算法生成，其判定规则如下：

- **TP**：该告警在后续版本中不再出现，算法认为它已被开发者修复，推测为真实问题
- **FP**：该告警在后续版本中仍然存在，算法认为开发者未予修复，推测为误报

> **⚠️ 重要提示**：该算法标签**仅供参考**，存在以下局限性：
> - 告警消失可能是代码重构、文件移动等原因，而非真正修复了安全问题
> - 告警持续存在可能是开发者尚未处理，而非代码本身安全
> - **请勿直接采信 `label`，应独立分析代码后再与 `label` 对比，若两者矛盾，应以代码分析结论为准**

### `sliced_code` 字段说明

`sliced_code` 是针对告警点进行数据流/控制流分析后提取的切片，已尽可能包含判断该告警所需的关键上下文。**请以切片代码中可见的信息为主要依据进行判断**，不应仅因切片之外可能存在代码而将告警标记为 Unknown。只有当切片代码本身确实缺少做出判断所必须的核心信息时，才考虑标记为 Unknown。

## 3. 分类标准

### ✅ TP（True Positive - 真实告警）

满足以下条件时判定为 TP：

- 代码中确实存在规则描述的安全漏洞或缺陷
- 该问题可能导致实际的安全风险或功能异常
- 需要开发者修复此问题

**判断要点**：

- 检查是否存在输入验证缺失、未授权访问、资源泄露等问题
- 分析代码执行路径，确认漏洞是否可被触发
- 考虑代码上下文，确认问题不是被其他机制防护

---

### ❌ FP（False Positive - 误报）

满足以下条件时判定为 FP：

- 工具报告了问题，但实际代码是安全的
- 存在其他防护机制（如前置校验、框架保护）使该告警无效
- 代码逻辑保证了不会触发该安全问题

**判断要点**：

- 检查是否有前置的输入校验或安全过滤
- 分析框架或库是否已提供了安全保护
- 确认代码路径在实际场景中是否可达

---

### ❓ Unknown（无法确定）

**仅在以下情况下**判定为 Unknown（需满足至少一条，且无法通过推断弥补）：

- 切片代码中关键变量的来源或赋值**完全缺失**，无法判断其是否受控（例如切片仅含使用点，而无任何赋值或传递路径）
- 告警涉及跨进程、跨服务、或依赖运行时外部配置的行为，且切片中无任何线索可供推断
- 告警规则本身需要调用链或接口契约信息，而切片中完全没有体现调用方的任何信息

**判断要点（判定 Unknown 前须逐条排除）**：

- 切片中是否有足够信息追踪关键变量的数据流？若有，则不应标为 Unknown
- 已可见的代码逻辑是否足以确认或排除漏洞？若足以，则应优先判定 TP 或 FP
- **不能仅以"切片之外可能存在防护"或"切片之外可能存在其他问题"为由判定 Unknown**，应基于切片内可见的证据做出倾向性判断

## 4. 分析步骤

1. **理解告警规则**：明确该规则检测的是什么类型的安全问题
2. **阅读代码上下文**：仔细分析切片代码，切片已覆盖与告警相关的关键数据流，应充分信任其完整性
3. **追踪数据流**：关注关键变量的来源、传递和使用，优先在切片内寻找证据
4. **评估防护措施**：检查切片中是否有验证、过滤或其他安全机制
5. **参考算法标签**：将 `label` 作为辅助信息，若与代码分析结论一致则增强信心，若矛盾则以代码分析为准
6. **做出判断**：优先根据切片内可见的证据判定 TP 或 FP；仅当切片内确实缺少做出判断所必须的核心信息时，才判定 Unknown

## 5. 输出要求

**严格遵守以下格式要求：**

- 输出必须是**纯 JSON 格式**
- 不得包含任何 Markdown 标记（如 ` ```json ` 代码块）
- 不得包含除 JSON 外的任何文本或解释
- 必须且仅包含以下三个字段：`id`、`llm_label`、`llm_label_reason`
- `llm_label` 的值只能是：`TP`、`FP` 或 `Unknown`
- `llm_label_reason` 应简明扼要地说明判定理由（1-3 句话）

**输出格式示例：**

```json
{
  "id": 1,
  "llm_label": "TP",
  "llm_label_reason": "代码未对用户输入进行SQL注入过滤，直接拼接到SQL查询中，存在明显的SQL注入风险。"
}
```
